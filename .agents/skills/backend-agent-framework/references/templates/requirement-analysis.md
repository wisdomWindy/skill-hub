# Requirement Analysis 模板

## 必需工件路径

- `docs/requests/<request-id>/requirements/requirement-analysis.md`

## `requirements/requirement-analysis.md`

必需章节：

- 资料来源与阅读口径
- 已确认口径
- 请求摘要
- 需求目标与业务价值
- 范围内事项
- 非目标与范围外事项
- 关键参与者与调用方
- 项目、服务与数据归属边界
- 显式约束归纳
- 隐含约束与假设
- 核心流程与状态分析
- 多模式/多角色差异流程
- 关键实体、接口与事件观察
- 外部系统与既有能力复用边界
- 验收与回归基线
- 研发拆分建议
- 风险、歧义与待确认项
- 拆分策略与建议拆分轴
- architecture-design 前置信号
- requirement-splitting 输入检查清单

### `资料来源与阅读口径`

至少覆盖：

- 原始需求来源、补充来源、引用资料和读取结果
- 采用哪份资料作为 source of truth
- 原型、PRD、接口文档、线下表格、画板、历史代码或项目文档冲突时的判断口径

### `已确认口径`

至少覆盖：

- 已经由产品、业务、研发确认的流程口径
- 已确认的非目标、不做项、灰度口径和发布口径
- 已确认的权限、状态、审核、同步、删除、幂等、回滚、隔离等关键规则

### `项目、服务与数据归属边界`

至少覆盖：

- 涉及哪些项目、服务、应用或仓库
- 每个项目/服务承担什么职责
- 主数据归属、跨库访问边界、同步边界和不应越界承接的能力

### `多模式/多角色差异流程`

至少覆盖：

- 不同账户类型、角色类型、制作模式或业务模式下的流程差异
- 各模式的创建入口、审核入口、终态和关键差异
- 哪些流程是主流程，哪些属于交付后独立申请或例外流程

### `外部系统与既有能力复用边界`

至少覆盖：

- 外部系统依赖、既有内部系统依赖和复用方式
- 哪些能力必须复用，哪些能力明确不在本次重建范围
- 对外接口待补时，允许以 adapter、配置和预留 DTO 继续推进的边界

### `验收与回归基线`

至少覆盖：

- 功能验收主线
- 不受影响范围和发布前必须回归的系统/链路
- 权限矩阵、状态机、文件边界、同步边界、外部依赖隔离等高风险回归面

### `研发拆分建议`

至少覆盖：

- 推荐按哪些研发模块推进
- 哪些模块是基础前置
- 哪些模块虽然可拆分，但发布验收仍必须按全链路整体通过

### `显式约束归纳`

至少覆盖：

- 接口、事件、字段、状态、错误码约束
- 权限、幂等、超时、重试、回调、迁移、回滚、观测性要求
- 上游文档明确写出的不可更改前提

### `隐含约束与假设`

至少覆盖：

- 默认读写路径或调用顺序假设
- 兼容性、灰度、历史数据或跨服务依赖假设
- 若假设错误将影响拆分、设计或实现的风险点

### `拆分策略与建议拆分轴`

至少说明：

- 为什么要按当前维度拆分，而不是按页面、接口数量或代码目录机械拆分
- 推荐拆分维度，例如 service、endpoint contract、job/workflow、consumer/subscriber、data model/schema、integration flow、cross-service rule
- 哪些内容必须保持在同一模块内，避免拆分后语义丢失

### `requirement-splitting 输入检查清单`

至少确认：

- 是否已具备稳定的范围边界
- 是否存在阻止无损拆分的关键歧义
- 哪些开放问题只需记录，哪些会阻止进入 requirement-splitting

### `风险、歧义与待确认项`

必须按以下分类列出：

- `业务阻塞`
  - 不确认会影响业务流程、数据模型、权限、状态机或审核口径，确认前不允许进入后续实现设计
- `外部接口待补`
  - 仅接口文档、字段语义、topic、签名或鉴权细节待补，允许按 adapter、配置和预留 DTO 先行设计
- `非阻塞`
  - 不影响主链路分析和后续拆分，可在设计评审或实现前补齐
